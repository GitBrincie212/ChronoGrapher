In our previous chapter, we have discussed the theory of monitoring and how important it is to get right, discussing
the various other solutions, but we haven't dived to how to actually use ``TaskHooks``.

Briefly restating ChronoGrapher's solutions, it uses ``TaskHooks``, they allow us to introspect at a deep level our Tasks,
while they can also satisfy multiple needs such as state management, markers... etc.

There are 2 phases to use the ``TaskHooks``, the former is the **Implementation / Definition Phase** while the
latter is **Registration Phase**, in the first, we explicitly declare something as a ``TaskHook``
and more specifically what events it can listen to.

Think of events as actions that happen in our code, such as retries, business logic doing (or have done) something
and even other TaskHooks triggering their own events.

## Implementation / Definition Phase
In the same previous code lets define our ``TaskHook`` like so:
<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/main.rs" lineNumbers=1
use chronographer::prelude::*;

#[derive(Default)]
pub struct MyCoolTaskHook;

#[hook]
impl MyCoolTaskHook {
    async fn OnTaskStart(&self, ctx: &TaskHookContext) {
        println!("Interested \"OnTaskStart\" event triggered!");
    }

    async fn OnTaskEnd(&self, ctx: &TaskHookContext, error: Option<&dyn TaskError>) {
        println!("Interested \"OnTaskEnd\" event triggered! With error = {:?}", error);
    }
}
```

<Callout title={"Requirement of \"async_trait\" for TaskHooks"} type={"warning"}>
    Everything related to TaskHooks requires a third party called ``async_trait``, simply install it and the
    ``#[hook(...)]`` macro takes care of utilizing it without explicitly including it for ergonomics.
</Callout>
</RenderProgrammingLanguageBased>
We observe how we defined our ``MyCoolTaskHook`` that acts as the TaskHook and <u>subscribed to two events, those being
    ``OnTaskStart`` and ``OnTaskEnd`` represented via the methods</u>. Both events are defined by ChronoGrapher which allow us to
listen to when a Task is about to start or end respectively.

Each method contains the code to execute when we listen to the event, or commonly called the
<Highlight color={"neutral"}>Side Effects</Highlight>, this method contains the ``self`` instance, a context object and
optionally a payload if the event itself includes it (you can avoid specifying it in the arguments even if it exists).

The ``self`` is just an instance of our ``TaskHook``, it works typically like any object. Whereas the context object gives us
detailed information about the event in question as well as an entrypoint to interact with the ``Task``.

Finally, the payload is extra metadata associated with the event, most event have their own payload which give detailed
information about what exactly happened. In the case of ``OnTaskStart``, its non-existent since no information is needed
whereas for ``OnTaskEnd`` it's needed to know whenever or not the ``Task`` failed and how.

With that said, in the first phase we have declared our TaskHook to listen and execute different code for the events
``OnTaskStart`` and ``OnTaskEnd``, we are not done, however.

## Registration Phase

The second phase or known as **Registration Phase** involves us attaching the ``TaskHook`` somewhere, more specifically
one of our Tasks as currently TaskHooks do nothing.

To achieve this we can change our code in the following way:
<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/main.rs" lineNumbers=28
#[task(schedule = every!(2s)))]
// [!code ++]
#[hook(auto(MyCoolTaskHook))]
async fn Task1(ctx: &TaskFrameContext) -> Result<(), TaskError> {
    println!("Simulating Task 1's Actions");
    Ok(())
}

#[task(schedule = every!(3s)))]
#[workflow(
    retry(3, 1s),
    timeout(5s)
)]
// [!code ++]
#[hook(
    // [!code ++]
    OnTaskStart: MyCoolTaskHook,
// [!code ++]
)]
async fn Task2(ctx: &TaskFrameContext) -> Result<(), TaskError> {
    println!("Simulating Task 2's Actions");
    Ok(())
}
```
</RenderProgrammingLanguageBased>
The above code automatically attaches our ``MyCoolTaskHook`` TaskHook to our Task instance (both task1 and task2) when we
instantiate.
<RenderProgrammingLanguageBased target_name={"Rust"}>
    <Callout title={"Context-Sensitive Nature Of The Macro"} type={"info"}>
        You may notice the ``#[hook(...)]`` macro is used in two different contexts, both generating the implementation
        of a ``TaskHook`` and registering said ``TaskHooks`` on various events of a ``Task``.

        Depending on the context of where it's used, it acts differently. The main rule of thumb is if its below a ``#[task(...)]``
        macro, its attaching. Whereas if its on top of an ``impl`` block (and soon we will see on top of a method) its defining.
    </Callout>

    For the former case, it utilizes ``auto(...)`` which allows to assign on demand our ``MyCoolTaskHook`` to multiple events
    based on what default events it specifies (also called <Highlight color={"primary"}>Auto-Attachment</Highlight>.
    Implicitly every event we define in our ``TaskHook`` is auto-attached by default.

    While, in the latter case, we assign explicitly and manually register those events to our ``MyCoolTaskHook``. The ``auto(...)`` acts
    as a shorthand to manually registering the above events (``OnTaskStart``, ``OnTaskEnd``).
</RenderProgrammingLanguageBased>

You may notice in the terminal the following:
```zsh title="terminal"
# After 2 seconds
Interested "OnTaskStart" event triggered!
Simulating Task 1's Actions
Interested "OnTaskEnd" event triggered! With error = None

# After 1 second
Interested "OnTaskStart" event triggered!
Simulating Task 2's Actions
```

Try removing the ``OnTaskEnd`` entry on ``task2``, notice how it no longer prints the ``OnTaskEnd`` contents?
<RenderProgrammingLanguageBased target_name={"Rust"}>
    Better yet, try to remove the entire ``#[hook(auto(MyCoolTaskHook))]`` line and see how it doesn't print neither the
    ``OnTaskStart`` nor ``OnTaskEnd`` for ``task1``.
</RenderProgrammingLanguageBased>

# Auto-Attachment & Gimmicks
While we briefly looked on how the ``#[hook(...)]`` macro at the top of the ``impl`` block can generate us an auto-attachment
method which automatically attaches all the events to our ``Task`` in the above example.

We haven't seen ways to configure the events (i.e. The defaults), this is where the third placement option for ``#[hook(...)]``
comes in picture. We can place this macro in individual methods inside the ``impl`` block of a parent ``#[hook(...)]``.

In this context, there are two parameters. The former is the ``default`` which controls which events are marked as default
and which aren't. We can modify our code to have as default the ``OnTaskEnd`` event attached:
```rust title="src/main.rs" lineNumbers=1
// <...>

#[hook]
impl MyCoolTaskHook {
    async fn OnTaskStart(&self, ctx: &TaskHookContext) {
        println!("Interested \"OnTaskStart\" event triggered!");
    }

    // [!code ++]
    #[hook(default)]
    async fn OnTaskEnd(&self, ctx: &TaskHookContext, error: Option<&dyn TaskError>) {
        println!("Interested \"OnTaskEnd\" event triggered! With error = {:?}", error);
    }
}

// <...>
```

Rerun the example and notice how the output changed to no longer including ``OnTaskStart``. Since we have specified a
default event, ChronoGrapher no longer assumes all events are default and only selects  ``OnTaskEnd`` as default.

There can be more events marked as default this way, however let's suppose a scenario with multiple events:
```rust title="src/main.rs" lineNumbers=1
// <...>

#[hook]
impl MyCoolTaskHook {
    async fn Event1(&self, ctx: &TaskHookContext) { /* ... */ }
    async fn Event2(&self, ctx: &TaskHookContext) { /* ... */ }
    async fn Event3(&self, ctx: &TaskHookContext) { /* ... */ }

    // ...

    async fn EventN(&self, ctx: &TaskHookContext) { /* ... */ }
}

// <...>
```

We want to mark as default everything except ``Event3`` but we don't want to append to every other event ``#[hook(default)]``
to discard it. To solve this we can utilize ``!default`` which discards the event as a default but lets through the rest marked
manually or automatically:
```rust title="src/main.rs" lineNumbers=1
// <...>

#[hook]
impl MyCoolTaskHook {
    async fn Event1(&self, ctx: &TaskHookContext) { /* ... */ }

    // [!code ++]
    #[hook(!default)]
    async fn Event2(&self, ctx: &TaskHookContext) { /* ... */ }

    async fn Event3(&self, ctx: &TaskHookContext) { /* ... */ }

    // ...

    async fn EventN(&self, ctx: &TaskHookContext) { /* ... */ }
}

// <...>
```

The latter parameter of the macro is ``listen``, its more of a cosmetic change (a different way of expressing the code).
Its use case is mainly for self-documenting code by allowing freedom of expression for the method's name as shown below:
```rust title="src/main.rs" lineNumbers=1
// <...>

#[hook]
impl MyCoolTaskHook {
    #[hook(listen = OnTaskStart)]
    async fn start_monitor_temporarily(&self, ctx: &TaskHookContext) { /* ... */ }

    // [!code ++]
    #[hook(listen = OnTaskEnd)]
    async fn shutdown_monitor_temporarily(&self, ctx: &TaskHookContext, error: Option<&dyn TaskError>) { /* ... */ }
}

// <...>
```

Lastly we return back to the parent ``#[hook(...)]`` macro at the top of the ``impl`` block. There is also a configuration for it
as well, specifically the auto-attachment. You can explicitly enable it via:
```rust title="src/main.rs" lineNumbers=1
// <...>

// [!code highlight]
#[hook(auto_attach)]
impl MyCoolTaskHook {
    // <...>
}

// <...>
```

Alternatively you can even override the name via, though you have to modify the various auto-attachments in the registration phase
(i.e. ``auto(...)``) to something like:
```rust title="src/main.rs" lineNumbers=1
// <...>

// [!code highlight]
#[hook(auto_attach = my_cool_auto_attach)]
impl MyCoolTaskHook {
    // <...>
}

#[task(schedule = every!(2s)))]
// [!code highlight]
#[hook(auto(MyCoolTaskHook, my_cool_auto_attach))]
async fn Task1(ctx: &TaskFrameContext) -> Result<(), TaskError> {
    // <...>
}
```

Or even fully disable it (again do note auto-attachment use cases will produce compiler errors):
```rust title="src/main.rs" lineNumbers=1
// <...>

// [!code highlight]
#[hook(!auto_attach)]
impl MyCoolTaskHook {
    // <...>
}

// <...>
```

The final gimmick of macro is you can have multiple ``impl`` blocks with different events. This allows not only for better
organization of events but the ability to group them to different auto-attachment methods (though you need to override each of their
names):
```rust title="src/main.rs" lineNumbers=1
// <...>

// [!code highlight]
#[hook(auto_attach = retry_autoattach)]
impl MyCoolTaskHook {
    // <...>
}

// [!code ++]
#[hook(auto_attach = tasks_autoattach)]
// [!code ++]
impl MyCoolTaskHook {
    // [!code ++]
    // <...>
// [!code ++]
}

// <...>
```

</RenderProgrammingLanguageBased>

# Stateful TaskHooks
TaskHooks aren't limited to behave as a pure function, in fact they can carry data with them which can be read and written
into (either from itself or from outside parties). As demonstrated below:
<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/main.rs" lineNumbers=1
use chronographer::prelude::*;
// [!code ++]
use std::sync::atomic::{AtomicUsize, Ordering}

// [!code highlight
// We use AtomicUsize to modify without the need of &mut self
// [!code highlight]
#[derive(Default)]
// [!code highlight]
pub struct MyCoolTaskHook {
    // [!code highlight]
    task_starts: AtomicUsize,
    // [!code highlight]
    task_ends: AtomicUsize,
// [!code highlight]
}

#[hook]
impl MyCoolTaskHook {
    async fn OnTaskStart(&self, ctx: &TaskHookContext) {
        println!("Interested \"OnTaskStart\" event triggered!");
        // [!code ++]
        self.task_starts.fetch_add(1, Ordering::Relaxed);
    }

    async fn OnTaskEnd(&self, ctx: &TaskHookContext, error: Option<&dyn TaskError>) {
        println!("Interested \"OnTaskEnd\" event triggered! With error = {:?}", error);
        // [!code ++]
        self.task_ends.fetch_add(1, Ordering::Relaxed);
    }
}
```
</RenderProgrammingLanguageBased>

And for our registration, let's modify it to utilize the manual approach (for guidebook reasons, which will become clear later):
<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/main.rs" lineNumbers=28
#[task(schedule = every!(2s)))]
#[hooks({
    // [!code highlight]
    OnTaskStart: MyCoolTaskHook::default(),
    // [!code highlight]
    OnTaskEnd: MyCoolTaskHook::default()
})]
async fn Task1(ctx: &TaskFrameContext) -> Result<(), TaskError> {
    // ...
}

// ...Same as above...
```
</RenderProgrammingLanguageBased>

However, we quickly stumble upon one problem, in our code when we subscribe the TaskHooks to each event, <u>we create a new
instance per event</u>, which means our metrics for tasks starting / ending will be inconsistent.

Try to define getter methods for the "metric values", then when the Tasks run, try to fetch it inside the main code
after a set amount of time:
<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/main.rs" lineNumbers
let task = Task1::instance();

// This method is used to fetch our MyCoolTaskHook from the event its subscribed
let hook = task.get_hook::<OnTaskStart, MyCoolTaskHook>().unwrap(); // You can use OnTaskEnd as well

let starts: usize = hook.get_start_count(); // Or whatever you want to call it
let ends: usize = hook.get_end_count(); // Again, whatever name you want
println!("VALUES ( {starts} | {ends} )");
```
</RenderProgrammingLanguageBased>
```zsh title="terminal"
# After 2 seconds
Interested "OnTaskStart" event triggered!
Simulating Task 1's Actions
Interested "OnTaskEnd" event triggered! With error = None

# After 1 second
Interested "OnTaskStart" event triggered!
Simulating Task 2's Actions

# When we fetch a TaskHook instance:
VALUES( 0 | 1 ) # Or the opposite with OnTaskEnd "VALUES( 1 | 0 )"
```

You will notice starts or ends is either zero, and only the opposite value is correct, to solve this we can define a
singleton / global TaskHook instance like so:
<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/main.rs" lineNumbers=28
// [!code ++]
use std::sync::LazyLock;
// [!code ++]

// [!code ++]
static MY_COOL_TASKHOOK_INSTANCE1: LazyLock<MyCoolTaskHook> = LazyLock::new(|| MyCoolTaskHook::default());

#[task(schedule = every!(2s)))]
#[hooks({
    // [!code highlight]
    OnTaskStart: MY_COOL_TASKHOOK_INSTANCE1,
    // [!code highlight]
    OnTaskEnd: MY_COOL_TASKHOOK_INSTANCE1
})]
async fn Task1(ctx: &TaskFrameContext) -> Result<(), TaskError> {
    // ...
}

// ...Same as above with a different instance...
```
</RenderProgrammingLanguageBased>
We define a singleton which contains the runtime state, we then attach the same TaskHook instance on multiple events without
creating new instances per event, resolving our mismatch issue.

<Callout title={"Singleton Multi-Task TaskHook Instance"} type={"idea"}>
Not only you can use a singleton TaskHook instance on multiple events per Task, but also on multiple Tasks at a time.
Though be careful in terms of how you expose this singleton in you application. You can limit the scope to a single Task
via:
<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/main.rs" lineNumbers=28
#[task(schedule = every!(2s)))]
#[hooks({
    // [!code ++]
    scopedSingleton = MyCoolTaskHook::default(),
    // [!code highlight]
    OnTaskStart: scopedSingleton,
    // [!code highlight]
    OnTaskEnd: scopedSingleton
})]
async fn Task1(ctx: &TaskFrameContext) -> Result<(), TaskError> {
    // ...
}

// ...Same as above with a different instance...
```
</RenderProgrammingLanguageBased>
</Callout>

<RenderProgrammingLanguageBased target_name={"Rust"}>
    Though for our simple code, we can keep ``auto(...)`` as it automatically assigns the same instance to multiple events.
    The reason we went with the manual approach over it is to showcase scenarios where ``auto(...)`` might not be possible to use.
</RenderProgrammingLanguageBased>

# Stateful Containers
While TaskHooks are nice for event listening, what happens when you only need to mount additional data onto a Task which
can be read / modified by outside parties only and not listen to any event.

This is where the **Stateful Container** pattern comes in. ChronoGrapher provides us with a special event called
**The Stateless Event** via <RenderProgrammingLanguageBased target_name={"Rust"}>``()`` and specifically ``TaskHook<()>``</RenderProgrammingLanguageBased>.

This event cannot be emitted no matter what and its intention is to be used for implementing TaskHooks with no
interested events, using it as simple as:
<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/main.rs" lineNumbers=28
// Implement TaskHook<()> and provide no code or optionally use the more
// idiomatic StatelessTaskHook trait alias for MyCoolTaskHook

#[task(schedule = every!(2s)))]
#[hooks({
    // ...

    (): MyCoolTaskHook::default(),

    // ...
})]
async fn Task1(ctx: &TaskFrameContext) -> Result<(), TaskError> {
    // ...
}

// ...Same as above with a different instance...
```
</RenderProgrammingLanguageBased>

We will be discussing how best to use them in the next chapter.

---

In short, TaskHooks have 2 phases, the **Implementation Phase** in which we implement all the necessary events for our
TaskHook to listen to and the **Registration Phase** where we assign a TaskHook instance to one or multiple events.

This can be done on one or multiple Tasks at a time. Singletons may be defined to help share state across events or Tasks
or both, you can scope them per Task or let them be global.

TaskHooks can also carry their own state which can be read / modified by itself or outside parties depending on how it wants
to expose its underlying data.

ChronoGrapher has a special event for the **Stateful Container** pattern via <RenderProgrammingLanguageBased target_name={"Rust"}>``()``</RenderProgrammingLanguageBased>,
it cannot be emitted no matter what, and is used for TaskHooks that aren't interested in any event but want to hold state.

we can make our TaskHook a stateful container with a special event.
