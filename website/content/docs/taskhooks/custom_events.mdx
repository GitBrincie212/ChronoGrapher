# Defining A Custom TaskHookEvent

Defining a custom event for our TaskHooks is as simple as:

<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/events.rs" lineNumbers
use chronographer::prelude::TaskHookEvent;

#[derive(Default)]
pub struct OnNotificationsSent;

impl TaskHookEvent for OnNotificationsSent {
    type Payload<'a> = usize where Self: 'a;
}

````
In our example above, we have defined `OnNotificationsSent` as a TaskHookEvent, by creating a typical struct and implementing the TaskHookEvent trait.
This trait requires the Default trait as such, we derived it.
In our previous chapter you may have noticed a payload parameter for our TaskHooks. This payload parameter is the same as the associated type Payload \<'a>. Here we set it to `usize` so listeners receive how many notifications were sent.
</RenderProgrammingLanguageBased>

Payloads are used to hand out extra information onto the TaskHook about the event, they are a transport layer between the cause of the event and the listeners (the TaskHooks).  
This data can be anything from a simple number to an array of strings to a handler... etc.

# Emitting Custom Events

Emiiting custom events from our defined task done like this:

<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/main.rs" lineNumbers
#[task(interval(10s))]
async fn DispatchNotificationsTask(ctx: &TaskFrameContext) -> Result<(), TaskError> {
    let sent = send_pending_notifications().await?;

    // [!code highlight]
    ctx.emit::<OnNotificationsSent>(&sent).await;

    Ok(())
}
```` 
In the example above we have a task called `DispatchNotificationsTask` that flushes pending notifications every 10 seconds. The `ctx` argument provided to every task is a `TaskFrameContext` reference which exposes the `emit` method. We use it to broadcast our custom event along with the number of notifications that were just sent.

</RenderProgrammingLanguageBased>

Now we have created custom event but we still lack the main component, which is listeners but before diving deep into the writing code. we need to understand one thing what happen if there is no listeners ?
so ChronoGrapher handles this without any overhead by simply doing no-op. 

You can implement listeners as follow: 

<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/main.rs" lineNumbers
use std::sync::atomic::{AtomicUsize, Ordering};

#[derive(Default)]
pub struct NotificationMetricsHook {
    total_sent: AtomicUsize,
}

#[async_trait]
impl TaskHook<OnNotificationsSent> for NotificationMetricsHook {
    async fn on_event(&self, _ctx: &TaskHookContext, payload: &<OnNotificationsSent as TaskHookEvent>::Payload<'*>) {
        let total = self.total_sent.fetch_add(*payload, Ordering::Relaxed) + *payload;
        println!("Dispatched: {} notifications | Running total: {total}", payload);
    }
}

````

```rust title="src/main.rs" lineNumbers
use std::sync::LazyLock;

static NOTIFICATION_METRICS: LazyLock<NotificationMetricsHook> = LazyLock::new(NotificationMetricsHook::default);

#[task(interval(10s))]
// [!code highlight]
#[hooks({ OnNotificationsSent: NOTIFICATION_METRICS })]
async fn DispatchNotificationsTask(ctx: &TaskFrameContext) -> Result<(), TaskError> {
    // ...
}
````

</RenderProgrammingLanguageBased>

Running this should produce output similar to:

```zsh title="terminal"
# After 10 seconds (50 notifications dispatched)
Dispatched: 50 notifications | Running total: 50

# After 20 seconds (1200 notifications dispatched)
Dispatched: 1200 notifications | Running total: 1250

# After 30 seconds (300 notifications dispatched)
Dispatched: 300 notifications | Running total: 1550
```

As you see output in terminal, we have successfully implemented receiver on our custom event. Now let's move ahead with Hook-To-Hook Communication.

# Hook-To-Hook Communication

Hook to Hook Communication will be done like this:

<RenderProgrammingLanguageBased target_name={"Rust"}>
```rust title="src/events.rs" lineNumbers
#[derive(Default)]
pub struct OnHighVolumeBurst;

impl TaskHookEvent for OnHighVolumeBurst {
    type Payload<'a> = usize where Self: 'a;
}

// Now update ``NotificationMetricsHook`` to emit ``OnHighVolumeBurst`` when a single dispatch is unusually large:

#[async_trait]
impl TaskHook<OnNotificationsSent> for NotificationMetricsHook {
    async fn on_event(
        &self,
        ctx: &TaskHookContext,
        payload: &<OnNotificationsSent as TaskHookEvent>::Payload<'_>
    ) {
        let total = self.total_sent.fetch_add(*payload, Ordering::Relaxed) + *payload;
        println!("Dispatched: {} notifications | Running total: {total}", payload);

        // [!code ++]
        if *payload > 1000 {
            // [!code ++]
            ctx.emit::<OnHighVolumeBurst>(payload).await;
        // [!code ++]
        }
    }
}

// Then define `AlertHook` and register it alongside `NotificationMetricsHook`:


#[derive(Default)]
pub struct AlertHook;

#[async_trait]
impl TaskHook<OnHighVolumeBurst> for AlertHook {
async fn on_event(&self, _ctx: &TaskHookContext, payload: &<OnHighVolumeBurst as TaskHookEvent>::Payload<'*>) {
    println!("ALERT: high-volume burst of {payload} notifications detected.");
}
}

#[task(interval(10s))]
#[hooks({
    OnNotificationsSent: NOTIFICATION_METRICS,
    // [!code ++]
    OnHighVolumeBurst: AlertHook::default(),
})]
async fn DispatchNotificationsTask(ctx: &TaskFrameContext) -> Result<(), TaskError> {
    // ...
}
````
As you go through the above code start understand the pattern like how we always define TaskHook and how Implementation for the that hook are always remain the same. the underlying principle remains the same.

</RenderProgrammingLanguageBased>

Running the full example now:

```zsh title="terminal"

# After 20 seconds (1200 notifications)
```
